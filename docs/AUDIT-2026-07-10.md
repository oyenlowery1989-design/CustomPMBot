# NoPMsBot v2 — Full Codebase Audit Report

**Date:** 2026-07-10 · **Scope:** all six audit dimensions (handler correctness, AI-drafted replies, database/migrations, security/access control, test suite, deployment/ops) · **All findings independently verified against the real code and the installed PTB 22.5.**

## Overview

After deduplication (the AI-callback auth gap and the unpinned-dependency problem were each independently confirmed by 2–3 reviewers), the audit yields **35 distinct findings**:

| Severity | Count |
|---|---|
| Critical | 3 |
| High | 8 |
| Medium | 11 |
| Low | 13 |

Plus one **verified-clean** result covering several deploy-safety risk areas (backup ordering, gitignore protection of `state.db`, single-instance handling, systemd hardening, DB path pairing).

**Overall risk assessment: HIGH.** The bot's core relay logic is sound in the happy path, but three critical defects are trivially triggerable in production today: (1) editing any command message crashes the handler, (2) the brand-new AI-draft feature shipped with **no authorization on any of its four callbacks** — anyone in the admin group can spend Anthropic API money or send fabricated replies to customers, and (3) a single slow Anthropic call (default read timeout: **600 seconds**) freezes the entire single-threaded bot for every user. A recurring architectural theme amplifies everything: the whole bot runs on one event loop with `concurrent_updates` off and every DB and AI call made synchronously, so any one slow path is a global outage. A second theme: the AI-drafted-replies feature (commit `dd7cf98`) accounts for 2 of 3 criticals and several highs/mediums — it needs a hardening pass as a unit. No audit dimension came back clean.

## Critical

### C1. Every command handler crashes on an edited command message
**`handlers/admin.py:21`** (plus user.py, wallet.py, topics.py, broadcast.py, tags.py, canned.py, autoreply.py, export.py, ai_reply.py — **107 call sites**) · crash

`CommandHandler` dispatches on `update.effective_message`, which includes `edited_message` updates, but the handler bodies read `update.message` directly. For an edited command, `update.message` is `None`, so `update.message.reply_text(...)` raises `AttributeError`. Reproduced against installed PTB 22.5. Any admin fixing a typo in `/ban 123` (or any user editing any command) crashes essentially every command in the bot.

**Fix:** Either (a) sweep all 107 call sites to `msg = update.effective_message`, or (b) — cheaper and safer — register command handlers with a filter excluding edits (`filters.UpdateType.MESSAGE`) so edited commands are simply ignored, then optionally do the sweep for defense in depth. Add one parametrized regression test that fires an edited-command update at every registered handler.

### C2. No authorization check on any AI-draft callback handler
**`handlers/ai_reply.py:53–142`** (`cb_ai_draft`, `cb_ai_send`, `cb_ai_edit`, `cb_ai_dismiss`) · auth bypass — *independently confirmed by three reviewers*

None of the four callbacks verifies `q.from_user.id in ADMIN_IDS` (or even that the callback came from `ADMIN_GROUP_ID`), unlike the sibling `cb_broadcast_confirm/cancel` at `handlers/broadcast.py:176/222` which do exactly that. Any non-admin group member who can see the buttons can trigger paid Anthropic API calls, **send a fabricated "reply" to a customer**, or dismiss/edit a pending draft. The test suite has `test_non_admin_ignored` only for `cmd_ai` — zero coverage for the callbacks, because there is nothing in the code to test.

**Fix:** Add the same `if q.from_user.id not in ADMIN_IDS: await q.answer(); return` guard used in broadcast.py to all four callbacks, plus a `rando_update()` negative test per callback (the helper already exists in `tests/test_handlers_ai_reply.py:75` — reuse it).

### C3. Synchronous Anthropic SDK call blocks the entire bot event loop
**`services/ai_draft.py:58`**, invoked at `handlers/ai_reply.py:77` · concurrency/error-handling

`generate_draft()` calls the **sync** Anthropic client inside an async handler with no executor offload and no timeout override — the client default is `Timeout(connect=5.0, read=600, ...)`. `bot.py:158` builds the Application without `.concurrent_updates()`, so updates are processed strictly serially: one slow/hung API call freezes the whole bot — all DMs, all admin commands, polling — for up to 10 minutes.

**Fix:** Use `anthropic.AsyncAnthropic` and `await` the call (or wrap the sync call in `asyncio.to_thread`), and pass an explicit timeout of ~30s with a caught-and-reported failure path. This must land with or before any wider rollout of the AI feature.

## High

### H1. Edited DMs and edited admin replies are silently reprocessed
**`bot.py:212`** · wrong-recipient / duplication

`handle_private_message` and `handle_admin_group_message` are `MessageHandler`s with no `UpdateType` filter, so they also fire for `edited_message` updates (verified with `check_update`). Editing a DM re-runs spam counting, wallet flows, topic creation and logging; editing an admin reply **re-sends the corrected text to the user as a brand-new message**, so the user gets both versions.

**Fix:** Add `filters.UpdateType.MESSAGE` to both handler registrations (ignore edits entirely), or handle edits explicitly by editing the relayed message.

### H2. Stale forum topic → messages from that user silently vanish forever
**`handlers/relay.py:28`** · data loss

`_ensure_topic()` trusts `users.topic_id` with no liveness check (unlike `_find_broadcast_topic`, which probes with `send_chat_action`). If a topic is deleted in Telegram, the forward's `TelegramError` is caught in `utils/media.py` and swallowed (`return None`) — topic_id is never cleared, nothing is recreated, no admin is notified. Every future message from that user is dropped.

**Fix:** On a forward failure whose error indicates a dead thread (e.g. "message thread not found"), clear `topic_id`, recreate the topic, retry once, and alert the admin group if recovery fails. Never swallow a relay failure silently.

### H3. AI "awaiting edit" intercept has no error handling — a send failure permanently jams the topic
**`handlers/relay.py:203`** · error-handling

Once a draft is `awaiting_edit`, every admin message in that topic is intercepted and sent via a bare `ctx.bot.send_message`. If that send raises (user blocked bot, >4096 chars), the status update never runs, so the draft stays `awaiting_edit` forever and **all future admin replies in that topic keep being swallowed and failing** — normal relay to that user is blocked until someone edits the DB by hand.

**Fix:** Wrap in try/except; on failure, notify the admin in-topic and either revert the draft to `pending` or mark it `failed` so the intercept releases. Truncate/validate length before sending.

### H4. Every DB call blocks the single event loop
**`database/connection.py:15`** · concurrency

All `database/*.py` functions make synchronous sqlite3 calls directly inside async handlers — zero `run_in_executor`/`to_thread` anywhere. Slow queries (LIKE search in `/search`, full scans, startup migrations) stall the entire bot including polling and background loops.

**Fix (pragmatic):** Wrap known-slow paths (search, analytics, migrations) in `asyncio.to_thread` first; longer-term, move to `aiosqlite` or a thin async DB facade. Pairs with H5.

### H5. `concurrent_updates` never enabled; a broadcast blocks all other updates
**`bot.py:158`** · concurrency

`Application.builder()` never calls `.concurrent_updates()`, so updates process one at a time. `MAX_CONCURRENT` only bounds the broadcast fan-out semaphore (`handlers/broadcast.py:79,123`) — a large broadcast run inside one handler invocation blocks every other incoming update until done.

**Fix:** Move broadcast fan-out into a background task (like the scheduled-broadcast loop), and/or enable `.concurrent_updates(N)` — but **only after** fixing the races it currently masks (M1, M2 below). Treat H4/H5/C3 as one "unblock the loop" workstream.

### H6. `cb_ai_draft` can orphan a DB row on send failure
**`handlers/ai_reply.py:87`** · error-handling

`db_create_draft(...)` precedes an unguarded `send_message` with no length truncation on the draft text. If the send fails (draft >4096 chars), the row is stuck `pending` with `topic_msg_id=NULL` forever, invisible to admins.

**Fix:** Truncate draft text to a safe length, wrap the send in try/except, and delete or mark-failed the row on failure with an in-topic error notice.

### H7. requirements.txt has zero version pins, contradicting the documented prod/tested split
**`requirements.txt:1`** · dependency — *confirmed by three reviewers*

No `==` versions on any of `python-telegram-bot / apscheduler / stellar-sdk / cryptography / httpx / anthropic`. Docs say prod runs PTB 20.7; only 22.x is installable today, and `utils/media.py:18-19` explicitly hedges code against exactly this drift. The very next `deploy.sh` run silently jumps two PTB major versions never validated in prod.

**Fix:** Pin every package (ideally `pip freeze`/pip-tools lockfile) to the versions actually validated; pin `requirements-dev.txt` too. Decide explicitly: standardize on 22.x (what tests run against) and update docs, or genuinely pin 20.7.

### H8. deploy.sh has no rollback if pip install fails mid-deploy
**`deploy/deploy.sh:58`** · deploy-safety

Service is stopped (lines 29–31) and the tree overwritten (41–56) **before** `pip install` (58–62). Under `set -euo pipefail`, an install failure (network blip, incompatible new release — likely, given H7) aborts the script with the bot fully down and nothing restarting it.

**Fix:** Install into the venv *before* stopping the service (or into a fresh venv swapped atomically), and add a trap/failure branch that restarts the previous service. Pinning (H7) removes most of the trigger surface.

### Verified clean
